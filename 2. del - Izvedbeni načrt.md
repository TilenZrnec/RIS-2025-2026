# Projektna dokumentacija - 2. del: Izvedbeni načrt 

## 1. Okviren projektni plan

Projekt bo trajal natanko **1 leto** (12 mesecev). Ekipo sestavlja **5 razvijalcev**, razvoj pa bo potekal po agilni metodologiji z uporabo 14-dnevnih sprintov (Scrum).

### 1.1 Razporeditev izdaj (Releases)

Projekt je razdeljen na 5 ključnih izdaj, ki zagotavljajo postopno uvajanje funkcionalnosti v okolje:

* **Izdaja 1 (3. mesec) - Temelj sistema in registracija:** Vzpostavitev baze podatkov in backend infrastrukture. Implementacija celotnega postopka registracije stranke (F-01, F-02, F-04) in priprave za pošiljanje kartic (F-05).
* **Izdaja 2 (6. mesec) - Logika lojalnosti in batch obdelava:** Razvoj batch storitev za uvoz podatkov v tabelo `Mesecni_nakup` (F-10) ter avtomatiziran izračun statusov (F-12, F-13) in točk (F-09, F-11) znotraj naše baze.
* **Izdaja 3 (9. mesec) - Uporabniški portal:** Razvoj spletnega vmesnika za stranke, ki omogoča neposreden vpogled v njihove podatke v bazi (točke, nakupi, katalog nagrad) in oddajo zahtevkov za koriščenje (F-16, F-17, F-18, F-19).
* **Izdaja 4 (11. mesec) - Administracijski modul:** Vzpostavitev vmesnika za upravljanje sistemskih pravil (F-14, F-15) in nadzor nad stanjem strank ter nagrad (F-20, F-22).
* **Izdaja 5 (12. mesec) - Optimizacija in končni zagon:** Testiranje zmogljivosti poizvedb nad 500.000 zapisi, finalizacija internacionalizacije in migracija v produkcijsko okolje.

### 1.2 Načrt prvih dveh sprintov

* **Sprint 1 (Vzpostavitev temeljev):**
    * Modeliranje in kreiranje Oracle baze po potrjenem ER diagramu (entitete `Stranka`, `Kartica_lojalnosti`, `Status`).
    * Vzpostavitev backend ogrodja (Node.js/Spring Boot) in povezave z bazo preko ORM plasti.
    * Razvoj API-ja za vnos nove stranke v bazo ob registraciji (F-01).
* **Sprint 2 (Varnost in Onboarding):**
    * Razvoj logike za verifikacijo e-naslova (posodobitev polja v tabeli `Stranka`).
    * Implementacija storitve za avtomatsko dodelitev začetnega statusa v tabelo `Zgodovina_statusa` (F-04).
    * Vzpostavitev avtentikacijskega mehanizma za varen dostop do podatkov stranke.

## 2. Okviren načrt testiranja

V nadaljevanju je okviren načrt testiranja za tri izbrane primere uporabe (PU), ki so že definirani v specifikaciji zahtev.

### 2.1 PU: Spletna registracija stranke (F-01)

**Glavni tok:**
1. Stranka odpre obrazec za registracijo.
2. Stranka vnese obvezne osebne podatke in potrdi registracijo.
3. Sistem validira podatke in ustvari zapis stranke.
4. Sistem pošlje verifikacijski e-mail.
5. Stranka klikne verifikacijsko povezavo.
6. Sistem aktivira račun, dodeli začetni status **osnovni** in sproži pošiljanje kartice lojalnosti.

**Stranski tokovi:**
- 3a (Stranski tok: Neveljavni ali manjkajoči obvezni podatki): Obvezni podatki manjkajo ali so v napačnem formatu -> sistem prikaže napake in ne shrani registracije.
- 4a (Stranski tok: Neuspešno pošiljanje verifikacijskega e-maila): Pošiljanje verifikacijskega e-maila ne uspe -> sistem registracijo označi kot neaktivno in sproži ponovni poizkus pošiljanja.
- 5a (Stranski tok: Potekla verifikacijska povezava): Verifikacijska povezava je potekla -> sistem zahteva ponovno pošiljanje verifikacije.

### 2.2 PU: Mesečni batch izračun točk (F-09, F-10, F-11, F-12, F-13)

**Glavni tok:**
1. Sistem ob planiranem terminu zažene mesečni batch proces.
2. Sistem iz poslovnega IS uvozi mesečne zneske nakupov.
3. Sistem za vsako aktivno stranko najprej posodobi status po pravilih prehajanja.
4. Sistem na podlagi novega statusa in točkovnika izračuna točke.
5. Sistem zapiše dodeljene točke in posodobi skupno stanje točk stranke.
6. Sistem zaključi batch obdelavo in pripravi poročilo za administratorja.

**Stranski tokovi:**
- 2a (Stranski tok: Neuspešen uvoz iz poslovnega IS): Uvoz podatkov iz poslovnega IS ni uspešen -> batch se ustavi, napaka se zabeleži, administrator prejme obvestilo.
- 3a (Stranski tok: Manjkajoči podatki o statusu stranke): Za posamezno stranko manjkajo podatki o statusu -> stranka se preskoči, primer se zabeleži v poročilu napak.
- 4a (Stranski tok: Neveljavna konfiguracija pravil točkovanja): Pravila točkovanja niso veljavno konfigurirana -> sistem ne dodeli točk za prizadete zapise in označi potreben ročni pregled.

### 2.3 PU: Koriščenje točk (F-19)

**Glavni tok:**
1. Prijavljena stranka odpre katalog nagrad.
2. Stranka izbere nagrado in potrdi koriščenje točk.
3. Sistem preveri razpoložljivo stanje točk.
4. Sistem odšteje ustrezno število točk in zapiše transakcijo koriščenja.
5. Sistem prikaže potrditev uspešnega koriščenja.

**Stranski tokovi:**
- 3a (Stranski tok: Premalo točk za izbrano nagrado): Stranka nima dovolj točk -> sistem zavrne koriščenje in prikaže trenutno stanje točk.
- 4a (Stranski tok: Napaka pri zapisu transakcije v bazo): Med potrditvijo pride do napake pri zapisu v bazo -> sistem transakcijo razveljavi in prikaže obvestilo o neuspehu.
- 4b (Stranski tok: Nagrada ni več aktivna): Izbrana nagrada med postopkom ni več aktivna -> sistem prekine postopek in zahteva ponovno izbiro nagrade.

## 3. Backend - Definicija API-jev

Vsi API dostopi so namenjeni komunikaciji med spletno aplikacijo (frontend) in našo centralno bazo podatkov. API-ji omogočajo branje in zapisovanje v entitete, definirane v podatkovnem modelu.

### 3.1 Upravljanje strank in dostopa

* **`POST /api/v1/stranka/registracija`** – Kreiranje novega zapisa v tabeli `Stranka`.
    * **Payload:**
        ```json
        {
          "ime": "Janez",
          "priimek": "Novak",
          "e_naslov": "janez.novak@maestro.si",
          "geslo_hash": "argon2$v=19$m=65536...",
          "ulica": "Slovenska cesta",
          "hisna_stevilka": "1a",
          "postna_stevilka": "1000",
          "kraj": "Ljubljana"
        }
        ```
* **`PATCH /api/v1/stranka/verifikacija`** – Posodobitev polja `status_verifikacije` v bazi.
    * **Payload:**
        ```json
        {
          "ID_stranke": 12345,
          "status_verifikacije": "verificiran"
        }
        ```
* **`POST /api/v1/auth/prijava`** – Preverjanje poverilnic v tabeli `Stranka`.
    * **Payload:**
        ```json
        {
          "e_naslov": "janez.novak@maestro.si",
          "geslo_hash": "..."
        }
        ```
    * **Response:** JWT žeton in osnovni podatki o stranki (`ID_stranke`, `ime`, `priimek`).
* **`PUT /api/v1/stranka/profil`** – Posodobitev osebnih podatkov v tabeli `Stranka` (F-08).
    * **Payload:**
        ```json
        {
          "ulica": "Nova ulica",
          "hisna_stevilka": "10",
          "postna_stevilka": "2000",
          "kraj": "Maribor"
        }
        ```

### 3.2 Podatki o zvestobi in transakcijah

* **`GET /api/v1/stranka/status`** – Pridobitev trenutnega statusa iz tabele `Zgodovina_statusa` (F-20).
    * **Response:**
        ```json
        {
          "naziv_statusa": "srebrni",
          "datum_od": "2023-01-01",
          "datum_do": null,
          "mesec": 1,
          "leto": 2023
        }
        ```
* **`GET /api/v1/stranka/tocke`** – Pridobitev polja `status_tock` iz tabele `Stranka` ter zgodovine transakcij.
    * **Response:**
        ```json
        {
          "status_tock": 450,
          "zgodovina_dodelitev": [
            { "mesec": 12, "leto": 2022, "stevilo_tock": 50, "status_ob_dodelitvi": "osnovni" }
          ],
          "zgodovina_koriscenja": [
            { "datum_koriscenja": "2023-01-15", "stevilo_porabljenih_tock": 100 }
          ]
        }
        ```
* **`GET /api/v1/stranka/nakupi`** – Seznam nakupov iz tabele `Mesecni_nakup` (F-17).
    * **Response:**
        ```json
        [
          { "mesec": 1, "leto": 2023, "znesek_eur": 540.50 },
          { "mesec": 12, "leto": 2022, "znesek_eur": 210.00 }
        ]
        ```

### 3.3 Nagradni program in katalog

* **`GET /api/v1/katalog/nagrade`** – Branje seznama nagrad iz tabele `Nagrada` (F-18).
    * **Response:**
        ```json
        [
          { "ID_nagrade": 1, "naziv": "Bon za 10€", "cena_v_tockah": 1000, "status_nagrade": "aktivna" }
        ]
        ```
* **`POST /api/v1/katalog/korisčenje`** – Zapis nove transakcije v tabelo `Korisčenje_tock` (F-19).
    * **Payload:**
        ```json
        {
          "ID_stranke": 12345,
          "ID_nagrade": 1,
          "stevilo_porabljenih_tock": 1000,
          "datum_koriscenja": "2023-04-15"
        }
        ```

### 3.4 Upravljanje sistemskih pravil (Administracija)

* **`GET /api/v1/admin/pravila-tockovanja`** – Seznam pravil iz tabele `Pravilo_tockovanja` (F-14).
    * **Response:**
        ```json
        [
          { "naziv_statusa": "osnovni", "znesek_od": 0, "znesek_do": 200, "stevilo_tock": 5 }
        ]
        ```
* **`PUT /api/v1/admin/pravila-tockovanja`** – Posodobitev zapisa v tabeli `Pravilo_tockovanja`.
    * **Payload:**
        ```json
        { "ID_pravila": 1, "stevilo_tock": 6, "datum_veljavnosti_do": "2023-12-31" }
        ```
* **`GET /api/v1/admin/pravila-statusa`** – Seznam pravil iz tabele `Pravilo-statusa` (F-15).
    * **Response:**
        ```json
        [
          { "tip_pravila": "napredovanje", "status_iz": "osnovni", "status_v": "srebrni", "prag_zneska_eur": 499 }
        ]
        ```
* **`PUT /api/v1/admin/pravila-statusa`** – Posodobitev zapisa v tabeli `Pravilo-statusa`.
    * **Payload:**
        ```json
        { "ID_pravila": 2, "prag_zneska_eur": 500 }
        ```
* **`GET /api/v1/admin/statistika`** – Agregirani podatki iz baze (F-21).
    * **Response:** Število strank po statusih, skupni znesek nakupov v določenem obdobju.
